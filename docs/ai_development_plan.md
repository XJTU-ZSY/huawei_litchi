# AI 全程开发作战方案

本文是教练、协议专家、架构师的首轮共识。后续实现、测试、回放分析都按本文推进。

## 总目标

用 Python 3.12 实现一个离线可提交的比赛客户端，在 3 天内形成可参赛、可复盘、可持续优化的策略程序。

胜率优先级：

1. 不掉线，每个 `inquire.round` 都回复 `action.round`。
2. 少非法动作，避免交付后违规。
3. 稳定完成 `S15` 交付。
4. 尽量达到普通皇榜任务基础分 90，释放送达基础分和用时分价值。
5. 根据回放优化路线、任务、资源、窗口和终局急策。
6. 从对手回放中抽取可复用策略。

## 角色工作流

每一轮迭代都走固定交接：

```text
litchi-coach
  -> 定目标、定优先级、给需求卡
litchi-protocol-expert
  -> 查任务书/协议，补动作、字段、合法性约束
litchi-architect
  -> 定模块、接口、测试边界
litchi-implementer
  -> 按需求卡写 Python 代码
litchi-tester
  -> 跑单测、假服务器、回放回归、打包检查
litchi-replay-analyst
  -> 赛后从回放提 bug、机会点、对手策略
```

任何回放发现都必须先变成需求卡，再进入实现。不要直接凭感觉改策略。

## 协议底线

协议专家确认以下规则必须先稳定：

- TCP 帧格式是 `5 位十进制长度前缀 + UTF-8 JSON body`。
- 接收端必须处理半包、粘包、中文 UTF-8 跨包和大消息。
- 消息流程是 `registration -> start -> ready -> inquire/action -> over`。
- 收到 `inquire.round = N` 后，只能发送 `action.round = N`。
- 没有动作也必须发送空动作心跳：`actions: []`。
- `start` 中的 `matchId`、玩家身份、地图、路线、资源、任务模板必须缓存。
- 不能写死玩家 ID、服务器地址、端口、阵营、地图路线、资源点或任务点。
- `actionResults.accepted = true` 不等于收益到账，必须结合 `events[]` 和下一帧状态确认。

## 目标架构

首版使用标准库，避免比赛运行时依赖安装。

```text
litchi_bot/
  __main__.py              # python -m litchi_bot 入口
  client.py                # TCP 客户端主循环
  protocol/
    framing.py             # 5 位长度前缀编解码
    messages.py            # registration/ready/action 构造
  core/
    models.py              # 轻量数据结构
    game_state.py          # start/inquire 到内部状态
    graph.py               # 地图图结构和路径
    scoring.py             # 分数估算
  strategy/
    base.py                # Strategy 接口
    rule_guard.py          # 动作合法性守门
    baseline.py            # 第一版保守策略
  replay/
    parser.py              # 回放读取
    analyzer.py            # bug/机会点报告
tests/
tools/
start.sh
```

关键边界：

- `protocol` 不理解策略，只保证收发正确。
- `core` 不打开 socket，只维护状态、地图、估分。
- `strategy` 只输入状态、输出动作候选。
- `rule_guard` 是最后一道闸，宁可空动作，也不提交明显非法动作。
- `replay` 复用 `core` 和 `strategy` 的判断逻辑，便于把实战问题转成回归测试。

## 首版策略框架

首版策略不是最优解，目标是稳定交付和低错误率。

策略分层：

1. 状态保护层：如果已交付，只发空动作或 `WAIT`。
2. 处理层：如果处于移动、处理、验核、强制通行等读条状态，按协议允许行为等待。
3. 必要流程层：到达固定处理站先 `PROCESS`，到 `S14` 且进入 `RUSH` 后 `VERIFY_GATE`，到 `S15` 后 `DELIVER`。
4. 路径层：根据 `start.edges[]` 找到当前节点到 `S14/S15` 的可行路径。
5. 任务层：优先选择顺路或低绕路的 30 分任务，目标是达到 90 分。
6. 资源层：优先顺路马类和冰鉴；不为低价值资源大幅绕路。
7. 窗口层：首版只做安全出牌或弃权，后续由回放优化。
8. 终局急策层：宫宴冲刺时优先保证验核和交付，急策只在明确可用时提交。

## 3 天节奏

第 1 天：

- 完成协议底座、客户端主循环、空动作心跳。
- 完成 `start/inquire` 最小状态解析。
- 完成图结构和最短路径。
- 完成可以抵达、验核、交付的保守策略。
- 完成假服务器和基础单测。

第 2 天：

- 完成固定处理点、任务处理、资源领取和基础动作守门。
- 建立任务分 60/90/110 的策略权重。
- 建立回放解析和首批回归用例。
- 根据实战回放修复非法动作和浪费帧。

第 3 天：

- 针对高频对手优化路径、任务、资源、窗口出牌。
- 加固异常状态、拒绝事件、交付后行为。
- 打包自检，锁定稳定版本。
- 只做小步收益优化，不做大范围重构。

## 需求优先队列

P0 必须完成：

- TCP framing 编解码。
- 注册、开局、ready、每帧 action、over 退出。
- 策略异常降级为空动作。
- 基础状态解析：自己、对手、当前位置、状态、任务、资源、事件。
- 图路径：从当前点到目标点。
- `start.sh` 可执行入口。
- 单测和假服务器 smoke test。

P1 强烈建议完成：

- 固定处理点动作。
- 宫宴冲刺、宫门验核、终点交付。
- 任务选择和 `CLAIM_TASK`。
- 顺路资源 `CLAIM_RESOURCE` / 马类和冰鉴 `USE_RESOURCE`。
- 动作守门员。
- 回放解析初版。

P2 回放驱动完成：

- 窗口出牌策略。
- 小分队探路、清障、增援、削弱。
- 设卡、攻坚、强制通行和悬赏。
- 天气绕路和处理加时优化。
- 对手策略学习。

## 验收门槛

每次实现完成必须说明：

- 改了哪些文件。
- 跑了哪些测试。
- 覆盖了哪张需求卡。
- 哪些风险还没覆盖。

首版可参赛最低验收：

- `python -m unittest discover -s tests` 通过。
- 假服务器能完成注册、start、ready、至少 3 个 inquire/action。
- 每个 action 的 `round` 和 `matchId` 正确。
- 策略抛异常时客户端仍然发送 `actions: []`。
- `start.sh <playerId> <host> <port>` 能启动。

## 回放闭环

用户提供回放后，执行：

1. 读取回放格式，必要时写最小解析器。
2. 提取动作拒绝、非法动作、缺动作、浪费帧、错失任务/资源/交付。
3. 对比对手在路径、任务、资源、窗口和急策上的优势。
4. 生成需求卡和回归样例。
5. 按一轮一目标实现。
6. 重新跑回归，确认该局问题不会复发。

回放结论分类：

- Bug：协议错误、字段错误、非法动作、拒绝动作、状态判断错、崩溃、缺心跳。
- 优化：可减少帧数、提高任务分、保鲜、保好果、提高窗口胜率。
- 假设：看起来有收益但需要更多回放验证。

## 当前首轮决策

第一轮不写复杂策略，先实现协议底座和最低可运行客户端。原因：

- 掉线或回合不匹配会直接失去比赛资格或大量收益。
- 所有策略都依赖 `start/inquire/events` 解析正确。
- 有了假服务器和回放入口，后续 AI 迭代才有可靠反馈。
